## Post 7 — "How the Brain Scanner Works"

**Day:** Tuesday | **Time:** 7:30 AM EST
**Type:** Technical architecture deep-dive

### Post Text

Yesterday I launched BizBrain OS. Today I want to show you how the brain scanner actually works under the hood.

When you install the plugin and run the initial scan, here's what happens in about 60 seconds:

Step 1 — Filesystem Discovery. The scanner walks your configured paths looking for signals: .git directories, package.json files, README.md, configuration files, environment patterns. It's not reading your code line by line. It's reading the metadata that describes your work.

Step 2 — Project Fingerprinting. Each discovered repo gets fingerprinted. Tech stack detection from package.json, requirements.txt, Cargo.toml, go.mod. Framework detection from config files. Status inference from git activity.

Step 3 — Entity Extraction. Names, organizations, relationships — pulled from READMEs, docs, and project configs. "Client: Acme Corp" in a README becomes a tracked entity in your knowledge brain.

Step 4 — Knowledge Graph Assembly. All of this feeds into structured Markdown and JSON files. One folder per project. One folder per entity. Cross-referenced relationships. Action items. History logs.

The key architectural decision: Markdown plus JSON instead of a database. At the scale of a solo dev or small team — hundreds to low thousands of records — flat files are fast enough, infinitely portable, work with git, and are human-readable. If BizBrain OS disappeared tomorrow, you'd have perfectly organized files.

No vendor lock-in. No data export needed. Just folders and files you already know how to use.

## Post 13 — "Building a Claude Code Plugin: What I Learned"

**Day:** Wednesday | **Time:** 8:00 AM EST
**Type:** Technical tutorial / educational

### Post Text

Claude Code's plugin system is one of the most underappreciated developer tools right now. Here's what I learned building a full plugin from scratch.

The plugin architecture is surprisingly elegant. You define skills, commands, agents, and hooks. Skills are the core capabilities. Commands are the entry points. Agents run autonomously. Hooks trigger on events like session start or tool use.

BizBrain OS has 15 skills, 11 commands, 2 agents, and 2 hooks. Here's the structure that emerged:

Skills are modular and self-contained. Each one has a clear responsibility. brain-bootstrap handles the initial scan. entity-management tracks people and companies. knowledge-management structures context. gsd-workflow manages project execution. They don't depend on each other.

Commands map to user actions. When someone types /brain, it routes to the brain-bootstrap skill. /entity routes to entity-management. Clean and predictable.

Agents are the long-running intelligence. Entity Watchdog monitors conversations. Brain Gateway provides access to the full knowledge base. They run in the background without explicit invocation.

Hooks are the glue. SessionStart injects context automatically. PostToolUse writes heartbeats so the brain stays current even if a session crashes.

The biggest challenge was cross-platform compatibility. Claude Code runs on macOS, Linux, and Windows. I wrote polyglot hook scripts that detect the OS and adapt. CMD to Git Bash on Windows, native bash on Unix.

Profiles were a late addition that proved valuable. A developer profile, content creator profile, consultant profile, agency profile, and personal profile — each activating different skill combinations.

If you're thinking about building a Claude Code plugin, the surface area is large but the patterns are learnable.

## Post 17 — "MCP Servers and Plugin Architecture"

**Day:** Tuesday | **Time:** 7:30 AM EST
**Type:** Technical deep-dive

### Post Text

The Claude Code plugin architecture has a concept most developers haven't explored yet: MCP servers inside plugins.

MCP (Model Context Protocol) is how Claude Code connects to external tools. Think of it as a standardized way to give the AI access to databases, APIs, file systems, and services.

BizBrain OS uses MCP connections for some of its most powerful features. Here's how the architecture works in practice.

The plugin core is skill-based. 15 skills that handle distinct capabilities: scanning, entity management, knowledge graphs, project tracking, and so on. Each skill is self-contained with clear inputs and outputs.

On top of skills, MCP servers provide live data connections. Need to check Supabase? MCP connection. Need to sync with Notion? MCP connection. Need to read Slack messages? MCP connection.

The elegant part: profiles. BizBrain OS ships with 5 profiles — developer, content creator, consultant, agency, personal. Each profile activates a different combination of skills and MCP connections. A content creator doesn't need the GSD workflow engine. A consultant doesn't need the content pipeline. Profiles keep things lean.

The architectural decision that mattered most: MCP connections are expensive in context. Each one costs 3K-20K tokens of context window. So BizBrain manages them dynamically. Enable what you need, disable what you don't. Subprocess delegation lets you run a one-off MCP task without keeping the connection permanently active.

This matters because context window is a finite resource. Every token spent on tool connections is a token not available for actual work. Managing that budget is a real engineering problem.

If you're building plugins, think carefully about your context budget. It's the constraint that shapes everything.
